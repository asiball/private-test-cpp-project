# ADR 0001: コンポーネントを独立・任意ビルド可能にする

- ステータス: Accepted
- 日付: 2026-05-31

## 背景（Context）

本リポジトリは学習用モノレポであり、読者が「興味のある部分だけ取り出して試す」ことが多い。
新たに I2C バス（`i2c-hal`）、ADS1115 センサー、GPIO 割り込み（`gpio`）を追加するにあたり、
**一部のコンポーネントを取り除いても、残りがそのままビルド・動作する**ことが求められた。

## 決定（Decision）

1. **各ペリフェラル / センサーを独立コンポーネントにする**
   - `spi-hal`（既存）, `i2c-hal`, `gpio` はそれぞれ独立した静的ライブラリ。互いに依存しない。
   - ADS1115 は `libsensor` 内の **独立ターゲット `ads1115`** とし、MCP3008 用の `sensor` とは分離する。

2. **トップレベル CMake は「存在するものだけ」ビルドする**
   - `add_subdirectory` を `if(EXISTS .../CMakeLists.txt)` で囲み、ディレクトリ削除に耐える。
   - `ads1115` は `i2c-hal` がある時のみ、`examples/ads1115_alert_demo` は `ads1115` と `gpio` の
     両方がある時のみビルドする。欠けていればスキップし、他は壊れない。

3. **共有ログマクロは `common/include/logger.hpp` に置く**
   - `i2c-hal` / `gpio` が `spi-hal` に依存せずにログ API（`LOGE` 等）を使えるようにする。

## トレードオフ（C 開発者にも馴染みのある論点）

`logger.hpp` は `spi-hal/include/` と `common/include/` に実質同じ内容が 2 つ存在する。
これは **DRY（重複排除）** と **独立性（疎結合）** のトレードオフであり、本プロジェクトでは
学習用途として「各コンポーネントが単体で完結する」独立性を優先した。

> C でも「共通ヘッダを 1 つに集約する」か「各モジュールに小さなコピーを置く」かは日常的な判断です。
> 集約するとビルド順・依存が増え、コピーすると同期漏れのリスクが出る。正解は文脈次第——
> ここでは "取り出しやすさ" を重視した、という意思決定の記録です。

将来 `spi-hal` 側も `common/include/logger.hpp` を参照するようにまとめれば重複は解消できるが、
既存の `spi-hal` を変更するリスクを避けて現状は据え置いている。

## 影響（Consequences）

- 利点: 任意のコンポーネントを削除/追加してもビルドが通る。CI も独立ステップで検証できる。
- 欠点: `logger.hpp` の重複。変更時は 2 ファイルの同期が必要（本 ADR で明示）。

## 検証

`i2c-hal` / `gpio` / `examples` を削除した状態でトップレベルビルドが成功し、
`spihal` / `libsensor`(MCP3008) / `device-ctl` が生成され、`ads1115` はスキップされることを確認済み。
